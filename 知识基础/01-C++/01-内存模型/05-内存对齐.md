## 内存对齐
### 什么是内存对齐：现代计算机中内存空间都是按照字节(byte)进行划分的，所以从理论上讲对于任何类型的变量访问都可以从任意地址开始，但是在实际情况中，在访问特定类型变量的时候经常在特定的内存地址访问，所以这就需要把各种类型数据按照一定的规则在空间上排列，而不是按照顺序一个接一个的排放，这种就称为内存对齐，内存对齐是指首地址对齐，而不是说每个变量大小对齐。

### 为何要有内存对齐：
- 有些CPU可以访问任意地址上的任意数据，而有些CPU只能特定地址访问数据，因此，不同硬件平台具有差异性，这样的代码就不具有可移植性。如果在编译时就将地址对齐，这样就具有平台可移植性了。
- CPU每次寻址都是要消费时间的，并且CPU 访问内存时，并不是逐个字节访问，而是以字长（word size）为单位访问，所以数据结构应该尽可能地在自然边界上对齐，如果访问未对齐的内存，处理器需要做两次内存访问，而对齐的内存访问仅需要一次访问，内存对齐后可以提升性能。

### 对齐系数
每个特定平台上的编译器都有自己的默认"对齐系数"，常用平台默认对齐系数如下：
- 32位平台：4字节对齐
- 64位平台：8字节对齐

### 结构体的内存对齐规则
- 对于结构体的各个成员，第一个成员位于偏移为0的位置，结构体第一个成员的偏移量(offset)为0，以后每个成员相对于结构体首地址的offset都是该成员大小与有效对齐值中较小那个的整数倍，如有需要编译器会在成员之间加上填充字节。
- 除了结构成员需要对齐，结构本身也需要对齐，结构的长度必须是编译器默认的对齐长度和成员中最长类型中最小的数据大小的倍数对齐。
- 成员变量的位置顺序也会影响结构体的内存布局
- 空结构体：
    - 空结构体的大小一般是1字节，这是为了防止空结构体的大小为0，导致的内存分配错误。 

### 手动设置对齐系数
- alignas是 C++11 引入的一个关键字，用于显式指定一个变量、类（或结构体）成员、类（或结构体）本身，或者一个动态分配内存块的对齐要求。
- 可以使用alignas()宏来手动设置对齐系数。
- 例如：alignas(32) int a;
- 这个a的内存对齐系数就是32字节。

- #pragma pack(n)
    ```C++
    #pragma pack(push, n)  // 将当前对齐设置压入“编译栈”，并设置新的对齐值为 n
    #pragma pack(n)         // 直接设置新的对齐值为 n（较不推荐，可能影响后续包含的头文件）
    #pragma pack(pop)       // 从“编译栈”弹出之前的对齐设置，恢复之
    #pragma pack()          // 恢复为编译器默认的对齐方式

    // 使用 #pragma pack(1) 进行1字节对齐（紧密排列）
    #pragma pack(push, 1) // 保存当前设置，并设置为1
    struct PackedAligned {
        char a;      // 偏移0
        int b;       // 偏移1 (min(4,1)=1的倍数，无需填充)
        short c;     // 偏移5 (min(2,1)=1的倍数)
        double d;    // 偏移7 (min(8,1)=1的倍数)
    }; // sizeof(PackedAligned) = 1 + 4 + 2 + 8 = 15
    #pragma pack(pop) // 恢复之前的对齐设置
    ```
- 主要应用场景：
    1. 节省内存空间：在内存极度受限的嵌入式系统中，通过 #pragma pack(1)消除填充字节，用空间换时间。
    2. 精确控制数据布局（最常见）：
        - 网络协议与通信：定义网络数据包、文件头或通信消息的结构时，必须确保发送方和接收方的内存布局完全一致，不受编译器和平台差异影响。通常使用 #pragma pack(1)。
        - 硬件寄存器映射：与特定硬件外设寄存器映射的结构体定义必须精确匹配硬件手册中描述的字节偏移，必须使用 pack 指令。
        - 与其他语言或旧二进制文件交互：确保C/C++结构体与来自其他语言（如某些特定格式的Delphi记录）或已存储的二进制文件的数据布局一致。

### 重要注意事项
- 平台与编译器差异：#pragma pack是大多数编译器（如 MSVC, GCC, Clang）都支持的非标准但事实通用的扩展。对于可移植性要求极高的代码，C++11 的 alignas和 alignof是更标准的选择，但 alignas通常用于增加对齐，而 pack常用于减少或取消填充，两者侧重点不同。
- 性能代价：使用 pack（特别是 pack(1)）可能导致未对齐的内存访问。在某些架构（如 ARM, SPARC）上，访问未对齐的数据会导致性能下降或直接引发硬件异常（总线错误）。在 x86/x64 架构上，通常能处理但仍有性能损失。
- 与 alignas的优先级：如果结构体成员有 alignas指定了更大的对齐值，它可能会覆盖 #pragma pack(n)设置的限制。编译器通常会取两者中较大的值作为该成员的实际对齐要求。
